## Context

当前 `harden.md` (322 行) 和 `SKILL.md` (187 行) 高度重复，交互流程为 Review → Plan → Execute → Report 四阶段，其中 Review 和 Plan 各自输出大量表格。用户反馈最痛的点是：表格连续轰炸、确认点被埋没、Execute 黑盒、两文件维护漂移。

## Goals / Non-Goals

**Goals:**
- 合并 Review + Plan 为单一 "评估与计划" 阶段，只输出风险摘要 + 执行计划概要
- 确认决策点紧跟在摘要后，确保首屏可见
- Execute 阶段每步输出 `[N/M]` 心跳，打破黑盒
- 最终报告结论先行，关键结果优先
- SKILL.md 瘦身为路由索引，harden.md 成为 SSOT
- BLOCKED 输出根据实际情境精简选项

**Non-Goals:**
- 不改变 harden 的核心能力范围（仍然是契约点驱动的测试加固）
- 不新增交互模式（如快速模式、增量模式）——本次只优化现有流程
- 不改变 Guardrails 约束（不改写 artifacts、不重构代码等）
- 不修改其他 opsx 命令

## Decisions

### D1: 摘要优先 + 按需展开 (Summary-first with drill-down)

**决策**: 顶层输出只包含核心状态（风险计数、涉及的模块名、验证范围概要），用户输入 "详情" 才展开完整契约点映射表。

**理由**: 95% 的场景中，用户在确认阶段只需要知道"有什么风险、要测什么模块"，不需要逐行审查契约点映射。详细数据在用户主动要求时再提供，避免信息过载。

**替代方案**: 始终输出完整表格但折叠——不如按需展开灵活，大表格仍然消耗 token。

### D2: Review + Plan 合并为单一阶段

**决策**: 将 Phase 2 (Review) 和 Phase 3 (Plan) 合并为 "Harden Assessment"，只有一个用户决策点。

**理由**: 两个阶段的输出目标用户是同一个人、同一个时刻。如果无 CRITICAL 风险，分阶段打断纯属无谓等待。合并后用户只需一次决策。

**边界**: 存在 CRITICAL 风险时，在评估阶段提前阻断，不进入决策点——因为 CRITICAL 需要用户介入解决。

### D3: 心跳反馈采用 `[N/M]` 格式

**决策**: Execute 阶段每步开始和结束时输出一行精简状态。

```
[1/4] 正在验证 grab 并发控制...
[1/4] ✓ 通过 (2 用例)
```

**理由**: 用户只需要知道"进度到哪了"和"有没有挂"，不需要完整构建日志。一行状态 + 结果标识足够。

### D4: 报告结论先行

**决策**: 报告第一行给出总体结果和关键数字，详情用精简表格补充。

```
## Harden 完成 — <name>
### 结果: ✓ 通过 (4/4 自动, 1/1 E2E) · 3 轮 · 0 待处理
```

**理由**: 6 张表格的原始报告用户难以快速判断"这次 harden 到底过没过"。结论先行让用户一眼看到结果，需要细节时往下翻。

### D5: SKILL.md 瘦身为路由索引

**决策**: SKILL.md 只保留 frontmatter 元信息 + 入口指向，所有业务逻辑收敛到 harden.md。

**理由**: 两文件 300+ 行重复内容，维护改一处忘一处。SKILL.md 作为 Skill 系统需要的注册文件保留最小骨架，harden.md 作为命令 Prompt 持有全部逻辑。

## Risks / Trade-offs

- **信息不足风险**: 摘要模式可能遗漏用户关心的细节 → 按需展开机制 (`输入"详情"`) 弥补
- **两文件一致性**: 瘦身后仍需确保 SKILL.md 的 frontmatter 与 harden.md 同步 → frontmatter 字段极少（name/description），漂移风险低
- **心跳输出过多**: 如果合约点很多（10+），心跳行可能刷屏 → 只输出步骤级心跳，不做子步骤报告
