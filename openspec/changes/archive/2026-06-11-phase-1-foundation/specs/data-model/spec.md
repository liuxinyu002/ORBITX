## ADDED Requirements

### Requirement: SCHEMA.md as authoritative data model document
The file `docs/SCHEMA.md` SHALL serve as the single source of truth for OrbitX's data model. It MUST contain:

1. A full entity-relationship diagram (ASCII art) showing all 5 tables and their relationships
2. Detailed column definitions with types, constraints, and business rules for each table
3. A complete index strategy table
4. A Phase-by-Phase migration SQL plan (V1 through V4)
5. UUID generation strategy specification
6. Timestamp format specification (ISO 8601 UTC)

#### Scenario: SCHEMA.md is complete and frozen
- **WHEN** a new developer joins the project
- **THEN** they can understand the full data model by reading only SCHEMA.md

### Requirement: Five-table ER model
The data model SHALL consist of exactly 5 tables:

1. **`app_kv`** — Application key-value store (Phase-1 physical)
2. **`model_configs`** — AI model provider configurations (Phase-2)
3. **`tasks`** — Data extraction tasks (Phase-3)
4. **`schema_fields`** — Per-task field definitions, 1:N from tasks (Phase-3)
5. **`extracted_data`** — Extracted structured data records (Phase-5)

#### Scenario: All 5 tables defined with complete specifications
- **WHEN** reading SCHEMA.md
- **THEN** each table has: column names, types, constraints, foreign keys, business rules

### Requirement: Phase-1 physical table constraint
Phase-1 SHALL create only the `app_kv` table physically. All other tables exist only as definitions in SCHEMA.md. The migration V1 SQL SHALL be:

```sql
CREATE TABLE app_kv (
    key         TEXT PRIMARY KEY,
    value       TEXT NOT NULL,
    created_at  TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
INSERT INTO app_kv (key, value) VALUES ('schema_version', '1');
INSERT INTO app_kv (key, value) VALUES ('ipc_status', 'ok');
```

#### Scenario: Phase-1 database contains only app_kv
- **WHEN** querying `SELECT name FROM sqlite_master WHERE type='table'` after Phase-1 migration
- **THEN** exactly one table `app_kv` exists (plus sqlite internal tables)

### Requirement: UUID v4 for all primary keys
All table primary keys (`id` columns) SHALL use UUID v4, generated by the Rust backend via the `uuid` crate. The frontend SHALL NOT generate IDs.

#### Scenario: Task creation generates UUID
- **WHEN** a new task is created via Tauri Command
- **THEN** the Rust backend generates a UUID v4 string
- **THEN** the ID is returned to the frontend

### Requirement: ISO 8601 UTC timestamps
All `created_at` and `updated_at` columns SHALL use ISO 8601 UTC format (`YYYY-MM-DDTHH:MM:SS.sssZ`). SQLite defaults SHALL use `datetime('now')` (UTC). Rust-side generation SHALL use `chrono::Utc::now().to_rfc3339()`.

#### Scenario: Timestamp stored in UTC
- **WHEN** a record is created
- **THEN** `created_at` contains an ISO 8601 UTC timestamp, not a local time

### Requirement: Foreign key cascade deletes
The following foreign key relationships SHALL use `ON DELETE CASCADE`:
- `schema_fields.task_id` → `tasks.id`
- `extracted_data.task_id` → `tasks.id`

This ensures that deleting a task automatically removes its schema fields and extracted data.

#### Scenario: Delete task cascades to schema_fields
- **WHEN** a task is deleted
- **THEN** all `schema_fields` rows referencing that task are automatically deleted
- **THEN** all `extracted_data` rows referencing that task are automatically deleted
