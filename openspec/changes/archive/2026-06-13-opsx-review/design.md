## Context

当前 OpenSpec 工作流为：`propose → apply → verify → archive`。没有方案评审环节。Claude Code 的 Skill 系统支持通过 SKILL.md 定义自定义 slash 命令，可用于扩展 OpenSpec 命令体系。

本项目是 Tauri 桌面应用（Rust + React），有严格的 DESIGN.md 规范、CLAUDE.md 施工准则、中文日志规范。评审命令需适配这些约束。

评审命令的详细需求已在 explore 阶段敲定，包括 6 个审核维度、3 级严重度、结构化诊断输出、具象化修复建议，以及“用户授权后直接修正原方案文件”的工作模式。

## Goals / Non-Goals

**Goals:**
- 实现 `/opsx:review` slash 命令，读取变更产物并输出结构化评审报告
- 评审覆盖 6 维度：执行/UI/架构/韧性/可观测性/依赖供应链
- 严重度分 3 级：🔴 Blocker / 🟡 Should Fix / 🔵 Suggestion
- 输出只展示有问题的维度；摘要区展示通过维度数量与下一步动作；Blocker 和 Warning 需附带具象修复建议
- 修复建议颗粒度控制在方案级（配置片段/架构伪代码/Schema 变更描述），严禁直接输出业务逻辑代码
- 未获授权时保持只读；获得用户明确授权后，最小化修改 proposal/design/specs/tasks 原文件
- 命令支持自动检测活跃变更，也接受显式变更名参数
- 多变更场景下的候选列表需降低选择成本，避免用户先手动打开多个 proposal 再决定
- 授权回写前需先展示拟修改文件和修改摘要，并允许用户按严重度或逐条控制修改范围

**Non-Goals:**
- 不修改既有 `/opsx:apply` 命令规则
- 不引入 `review.md` 之类的新状态文件
- 不覆盖测试策略审核（由 `/opsx:harden` 单独负责）
- 不修改 OpenSpec 核心系统（仅通过 Skill 体系扩展）

## Decisions

### 1. 实现方式：Claude Code Skill

**选择**: 通过 `.claude/skills/openspec-review/SKILL.md` 定义 `/opsx:review` 命令。

**替代方案考虑**:
- 修改 Claude Code 源码内置命令：不可行，Claude Code 是封闭源码
- MCP Server 扩展：过度工程化，Skill 体系已足够

Skill 的 System Prompt 将包含：
- 完整的 6 维审核检查清单（每项的检查逻辑和严重度）
- 强制输出格式模版（Few-Shot 标准示例）
- 修复建议颗粒度约束
- 默认只读 + 授权后可回写原方案文件的行为约束
- 摘要区下一步动作提示与授权粒度选择约束

### 2. 审核引擎：6 维度并行审查

**选择**: 在单次对话中依序执行 6 个维度的审查，而非并行启动 6 个子 Agent。

**理由**:
- 审查维度之间存在逻辑关联（如架构审核发现的 N+1 问题也会触发韧性审核的降级建议）
- 跨维度关联分析比并行速度更重要
- 单次对话上下文内各维度结果自然汇聚，无需事后合并

**6 维度清单及其检查逻辑见 SKILL.md Prompt 部分。**

### 3. 输出格式：强制结构化模板 + Few-Shot

**选择**: 在 System Prompt 中嵌入完整的 Few-Shot 输出示例，强制锁定 Markdown 结构。

**格式约束**:
```markdown
## 审查报告: <change-name>

**诊断完成：发现 X 个 Blocker，Y 个 Warning，Z 个维度通过。当前状态：🔴 需修正 / 🟢 可继续**

**下一步**：授权我修正方案 / 仅修复 Blocker / 修复 Blocker + Warning / 逐条确认 / 仅保留诊断

### <维度名>（N <severity>）
- [ ] <severity> <检查项描述>

**修复建议**：
```<lang>
<方案级代码/配置/伪代码>
```
```

**折叠规则**: 仅展示含问题的维度；通过的维度不逐条展开，只在摘要中显示通过数量。

### 4. 修复建议颗粒度控制

**选择**: 在 System Prompt 中设置硬约束——修复建议只能输出以下三类内容：
1. **配置片段**: `tauri.conf.json`、`capabilities/*.json`、`Cargo.toml` 等配置文件的精确变更
2. **架构伪代码**: trait 签名、enum 变体定义、数据流伪代码（不包含完整实现）
3. **Schema 变更描述**: 数据库表结构变更的 DDL 描述

**严禁输出**:
- 完整的函数体实现
- 前端组件的完整 JSX/TSX
- 业务逻辑的逐行代码

**理由**: 评审阶段定位为"只改图纸，不砌砖"。输出业务逻辑代码会污染评审的客观性，并鼓励开发者跳过方案修正直接复制代码。

### 5. 授权回写机制

**选择**: `/opsx:review` 默认仅输出诊断结果；当用户明确授权“按建议修改方案”后，先给出修改预览，再由同一 Skill 将最小修复建议直接回写到原方案文件。

**回写逻辑**:
1. 确定目标变更名（从用户输入或自动检测）
2. 读取 proposal/design/specs/tasks，完成 6 维评审并输出结构化问题清单
3. 若用户未授权修改 → 停在诊断阶段，不改任何文件
4. 若用户明确授权修改 → 先输出“拟修改文件 + 每个文件的最小修改摘要”
5. 用户选择修改范围：
   - 仅修复 Blocker
   - 修复 Blocker + Warning
   - 逐条确认
6. 仅修改与用户选定问题直接相关的方案文件
7. 修改范围仅限：
   - `openspec/changes/<name>/proposal.md`
   - `openspec/changes/<name>/design.md`
   - `openspec/changes/<name>/tasks.md`
   - `openspec/changes/<name>/specs/**`
8. 修改完成后输出变更回执，说明：
   - 已修改的文件
   - 每个文件对应修正了哪些问题
   - 哪些问题仍保留未改

**约束**:
- 不新增 `review.md` 或其他中间状态文件
- 不修改 `.claude/skills/openspec-apply-change/SKILL.md`
- 不在未获授权的情况下修改任何方案文件
- 未展示修改预览前，不得直接落盘

### 6. 活跃变更自动检测

**选择**: 检索 `openspec/changes/` 下排除 `archive/` 的所有子目录，若仅有一个活跃变更则自动选中；若有多个则展示候选列表并提示用户指定变更名。

**候选列表内容**:
- 变更名
- 最近修改时间
- proposal 的一句话摘要（优先取 Why/What Changes 首条）

**理由**:
- 降低多变更场景下的选择成本
- 避免用户为了确认目标变更而手动逐个打开 proposal

## Risks / Trade-offs

- **[Skill 长度]** 6 维度完整审查清单会显著增长 System Prompt（预估 ~300 行）。→ 审核维度精简为检查点形式，每条 1-2 行，避免冗长的理论阐述。
- **[授权边界]** 如果用户只想看诊断，不希望自动落盘，review 不能越权修改。→ 默认只读，仅在明确授权后回写原方案文件。
- **[修改扩散]** 授权回写时若一次性修改过多文件，会偏离“最小可行修复”。→ 每个问题只修改直接相关的 artifact，不顺手重写无关段落。
- **[交互冗长]** 如果每个通过维度都输出一行“✅ 通过”，报告会变长且噪音偏高。→ 只在摘要区汇总通过维度数量，不逐条展示。
- **[授权误操作]** 若用户一条“同意修改”就触发全部建议落盘，容易超出预期。→ 授权前先展示修改预览，并提供按严重度/逐条确认的粒度控制。
- **[全绿误判]** 若审核遗漏问题，report 显示全绿通过会制造虚假安全感。→ 6 维度检查清单需持续迭代，并在 Prompt 中强调"宁可多报 Warning，不可漏报 Blocker"。
