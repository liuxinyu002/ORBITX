## ADDED Requirements

### Requirement: model_configs table (Phase-2 physical table)
The database SHALL contain the `model_configs` table, created by migration V2:

```sql
CREATE TABLE model_configs (
    id          TEXT PRIMARY KEY,
    provider    TEXT NOT NULL,
    label       TEXT NOT NULL,
    base_url    TEXT NOT NULL,
    model_id    TEXT NOT NULL,
    model_name  TEXT NOT NULL,
    api_key     TEXT NOT NULL,
    is_active   INTEGER NOT NULL DEFAULT 0,
    created_at  TEXT NOT NULL DEFAULT (datetime('now')),
    updated_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
```

#### Scenario: model_configs table created by V2 migration
- **WHEN** migration V2 is executed
- **THEN** `model_configs` table exists with all columns defined above

#### Scenario: UUID v4 primary key
- **WHEN** a new model config is created
- **THEN** its `id` is a UUID v4 string generated by the Rust backend

#### Scenario: Single active model constraint
- **WHEN** a model config is set as active (`is_active = 1`)
- **THEN** all other model configs have `is_active` set to 0
- **THEN** the constraint is enforced in the Rust command handler via a transaction

#### Scenario: Modified data-model specification
- **WHEN** reading SCHEMA.md
- **THEN** the `model_configs` table specification includes all columns, types, constraints, and business rules matching the V2 migration

### Requirement: ModelConfig Rust struct
The Rust crate SHALL define a `ModelConfig` struct with serde Serialize/Deserialize:

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ModelConfig {
    pub id: String,
    pub provider: String,
    pub label: String,
    pub base_url: String,
    pub model_id: String,
    pub model_name: String,
    pub api_key: String,
    pub is_active: bool,
    pub created_at: String,
    pub updated_at: String,
}
```

#### Scenario: ModelConfig serializes to JSON
- **WHEN** a `ModelConfig` instance is serialized to JSON
- **THEN** all fields are included with correct types (is_active as bool)

### Requirement: ModelConfigInput for create/update operations
The Rust crate SHALL define a `ModelConfigInput` struct for create/update Tauri Commands. This struct SHALL NOT include `id`, `is_active`, `created_at`, `updated_at` fields (those are managed by the backend).

#### Scenario: ModelConfigInput excludes server-managed fields
- **WHEN** creating a new model config
- **THEN** the frontend sends only user-provided fields (provider, label, base_url, model_id, model_name, api_key)
- **THEN** the backend generates id, is_active default, and timestamps
