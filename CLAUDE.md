# CLAUDE.md

## 1. 编码前先思考

**不要假设。不要掩盖困惑。呈现取舍。**

实施前：
- 明确陈述假设。不确定就问。
- 存在多种理解时，全部列出来——不要默默选一个。
- 有更简单的方案就说出来。必要时据理力争。
- 不清楚就停下来，指出困惑之处，然后问。
- 不要恭维我。不要说"好问题""你说得很有道理""这是一个很好的思路"这类话。直接回答。 
- 如果你觉得我说的有问题，直接指出来，不要委婉地绕弯子。
- 如果我的想法本身就有明显的问题，不要帮我优化它，先告诉我问题在哪。

## 2. 简洁优先

**用最少的代码解决问题。不做任何推测性设计。**

- 不添加未要求的功能。
- 不为单次使用创建抽象。
- 不添加未要求的"灵活性"或"可配置性"。
- 不为不可能发生的场景写错误处理。
- 如果你写了 200 行但 50 行就能搞定，重写它。

问自己："资深工程师会觉得这过度工程化了吗？" 如果是，就简化。

## 3. 精准修改

**只动必须动的。只清理自己留下的。**

编辑已有代码时：
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没坏的东西。
- 匹配已有风格，即使你习惯不同。
- 发现无关的废弃代码，提出来——不要删掉它。

你的修改产生孤立代码时：
- 删除因你的修改而变得未使用的 import/变量/函数。
- 不要删除早已存在的废弃代码，除非被要求。

检验标准：每一行改动都应能追溯到用户的需求。

## 4. 日志使用中文

**日志输出点需在方案讨论时与用户一起确定，避免冗余日志。日志消息优先使用中文。**

前端日志通过 `log(level, target, message)` 函数桥接到 Rust 后端统一输出（`src/lib/logger.ts` → `log_event` Tauri command）。日志级别 `level` 取 `"error"` / `"warn"` / `"info"` / `"debug"`，`target` 为模块标识，`message` 为中文描述。如需附带结构化数据，将数据 JSON.stringify 后拼入 `message`。

示例：`log("info", "browser", "浏览器会话已启动")` 而非 `log("info", "browser", "browser session started")`。


## 5. 环境变量管理

**新增可调参数时，同步更新 `.env.example`。**

项目通过 `.env` 管理运行时可调参数（如 `LOG_LEVEL`、`NODE_ENV`）。每次新增环境变量时，必须在 `.env.example` 中添加对应示例条目并附注释说明用途与可选值。`.env` 已被 `.gitignore` 排除，不得提交。

## 6. 目标驱动执行

**定义成功标准。循环直至验证通过。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入写测试，然后让测试通过"
- "修复 Bug" → "写一个能复现它的测试，然后让测试通过"
- "重构 X" → "确保重构前后测试都通过"

多步骤任务，列出简要计划：
```
1. [步骤] → 验证：[检查方式]
2. [步骤] → 验证：[检查方式]
3. [步骤] → 验证：[检查方式]
```

强成功标准让你能独立循环。弱标准（"让它能用"）需要不断确认。

---

**准则有效的标志：** diff 中不必要的改动更少，过度工程化导致的重写更少，澄清问题在实施前而非出错后才出现。
